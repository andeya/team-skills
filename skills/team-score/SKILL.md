# AI 协作项目评分 Skill

你是 AI 协作项目的评委。根据以下评分标准，扫描当前项目的源码、文档、测试、配置文件等证据，对项目进行逐项评分，输出结构化评分报告和改进建议。

## 硬门槛（任一不通过则整体不通过）

在评分前先检查以下 7 项硬门槛，任一未通过则标记为「不通过」并说明原因：

| # | 硬门槛 | 验收方式 | 通过标准 | 不通过表现 |
|---|--------|---------|---------|----------|
| 1 | 提交 AI 任务规划 | 查看提交材料中是否有任务规划文件 | 至少包含目标、上下文、任务拆分、修改范围、验证方式 | 只有代码或聊天记录，没有任务规划文档 |
| 2 | 明确 AI 可改/不可改边界 | 检查任务规划或 Prompt | 明确写出允许修改文件、禁止修改文件、接口/数据/依赖约束 | AI 改了无关文件，没有设定边界 |
| 3 | 补充测试或验证手段 | 查看测试代码、测试记录、验证脚本 | 有新增、修改验证代码，或有前置验证 | 只有最终提交，没有过程验证 |
| 4 | 代码修复通过预设测试 | 运行预设测试或查看现场演示 | 指定演示全部通过 | 失败的测试或功能未通过 |
| 5 | AI 协作资产不是空口号 | 查看 AGENTS.md / CLAUDE.md / Checklist 等 | 规则可执行、可拆分、能指导后续任务 | AI 生成「代码规范」类空话，没有可执行规则 |
| 6 | 说明风险、新增问题和风险 | 查看交付说明 | 明确列出、验证计入；未覆盖内容有说明 | 未提及任何风险或未覆盖事项 |
| 7 | 能解释关键决策 | 书面答辩 | 能解释为什么这样设计测试，为什么采纳/拒绝 AI 建议 | 无法说明业务决策原因 |

## 交付物清单

评分前先核对以下三类交付物是否齐全：

### 1. AI 协作资产
- AGENTS.md / CLAUDE.md / Copilot Instructions / Cursor Rules 任一种或多种
- 业务术语表
- 项目/模块结构说明
- 编码规范
- 测试规范
- Review Checklist
- Delivery Checklist

### 2. AI 协作任务规划
- 目标澄清
- 上下文选择
- 任务拆分
- 文件修改边界
- 验证计划
- 风险识别
- 停下来问人的条件
- 给 AI 的最终任务提示词

### 3. 质量保障产物
- SDD 规格说明
- 测试用例矩阵
- 新增/修改的测试代码
- 缺陷修复代码
- 测试运行结果
- Review 发现与修复说明

## 评分维度（总分 100 分）

### 一、AI 协作资产沉淀（25 分）

| 二级验收项 | 分值 | 验收标准 | 需检查的证据 |
|-----------|------|---------|------------|
| 分层组织清晰 | 5 | 能区分项目级、模块级、任务级规则；没有把所有规则混在一个大文件里 | AGENTS.md / CLAUDE.md / Cursor Rules / Copilot Instructions / docs/ai/* |
| 内容覆盖完整 | 5 | 覆盖业务术语、系统架构、代码结构、接口约定、编码规范、测试要求、Review 标准、交付要求 | 业务术语表、架构说明、接口约定、测试规范、Review Checklist |
| 规则可执行 | 5 | 规则具体明确，不是「代码要优雅」这类空话；能指导 AI 后续执行 | 明确的必选项、禁止项、示例、编码方式 |
| 适配工具产物 | 5 | 至少产出 2 类工具适配产物（如 AGENTS.md、CLAUDE.md、Copilot Instructions、Cursor Rules、Prompt 模板、检查清单） | 工具规划文件、Prompt 模板、Checklist |

### 三、AI 交付质量保障（27 分）

## 执行流程

### Step 1: 收集证据

使用 subagent 并行扫描以下 5 个维度的证据来源：

**Agent 1 — AI 协作资产扫描**
- 查找所有 CLAUDE.md / AGENTS.md / .cursorrules / .cursor/rules / .copilot-instructions.md / docs/ai/* 文件
- 查找任务级规则文件：docs/tasks/*/task-rules.md
- 检查内容是否分层（项目级 vs 模块级 vs 任务级）
- 检查是否覆盖：业务术语表、系统架构（AGENTS.md / docs/architecture.md）、代码结构（AGENTS.md / CLAUDE.md）、接口约定、编码规范、测试规范、Review Checklist、Delivery Checklist、交付要求
- 检查规则是否具体可执行（有无禁止项、必选项、示例、编码方式）
- 检查有无 Prompt 模板（docs/tasks/*/prompt-template.md）、检查清单等工具适配产物（≥ 2 类）
- 检查有无维护说明、版本记录、集中新增规则机制（CLAUDE.md 中的「资产维护机制」段落）

**Agent 2 — 任务规划扫描**
- 查找 docs/ 下的任务规划、PRD、设计文档、实现计划
- 检查是否有目标澄清（成功标准、非目标）
- 检查是否有上下文选择说明（必要文件清单、排除了不必要上下文）
- 检查是否有分阶段执行计划（探索→方案→实现→验证→总结）
- 检查是否有修改文件边界说明（allow/deny）
- 检查是否有验证计划（每一步有可检查完成标准）
- 检查是否有风险识别
- 检查是否有停下来问人的条件
- 检查是否有给 AI 的最终任务提示词（独立的 prompt-template.md 或文档内嵌 Prompt）

**Agent 3 — 质量保障扫描**
- 检查 tests/ 目录结构和测试文件数量
- 检查是否有 SDD 规格文档（输入/输出/边界条件/异常场景/验收标准）
- 检查 git log 中的测试提交历史（是否有先写测试再修复的 TDD 模式：test: 提交 → feat:/fix: 提交）
- 检查测试覆盖范围（happy path / 边界 / 异常 / 回归）
- 检查测试用例矩阵文档
- 检查是否有缺陷修复代码及其测试验证（修复未破坏已有功能）
- 检查是否有 Review 记录（含脆弱性、数据、安全、性能、兼容性维度）和风险识别文档
- 检查测试运行结果记录
- 运行项目测试命令列出测试用例（如 `bun test --list`、`cargo test -- --list`、`pytest --collect-only` 等，根据项目技术栈选择）
- 检查 CI 配置（ci:fix、check 等）

**Agent 4 — 使用过程扫描**
- 查找 AI 对话记录、Prompt 记录（07-prompt-log.md）、纠偏记录
- 查找个人复盘文档（13-retrospective.md），检查是否有「本次沉淀的新规则」段落
- 检查 skills 和配置目录
- 检查是否有结构化的 Prompt（五要素：目标+上下文+边界+输出格式+验证标准）

**Agent 5 — 团队协作扫描**
- 检查 git log --format='%an' 统计贡献者
- 查找分工说明文档（需求澄清、AI 操作、测试、Review 等角色分工）
- 查找交叉 Review 记录（Review 是否发现真实问题，不只是格式建议）
- 检查产物风格一致性（多个文件写作风格统一，没有简单拼接）
- 检查每个人是否有独立提交物、Review 记录或复盘说明

### Step 2: 硬门槛判定

根据收集的证据，逐条判定 7 项硬门槛是否通过。任一不通过则在报告中醒目标注。

### Step 3: 逐项评分

对每个二级验收项：
1. 列出找到的证据（文件路径 + 关键内容摘要）
2. 根据验收标准给出得分（0 ~ 满分）
3. 给出简短评语

评分档次参考：
- **满分**: 完全符合验收标准，证据充分
- **80%**: 基本符合，有小缺失
- **60%**: 部分符合，有明显缺失
- **40%**: 有意识但执行不足
- **0分**: 完全缺失

```
| ... |

### 二、AI 协作任务规划（得分/25）
...

### 三、AI 交付质量保障（得分/27）
...

...

- 90-100: 优秀 — AI 协作能力成熟
- 75-89: 良好 — 有体系但需完善
- 60-74: 合格 — 基本意识具备但实践不足
- <60: 不合格 — 需要系统学习 AI 协作方法

1. ...
2. ...
```

### Step 5: 改进建议

改进建议直接在对话中输出，不写入文件。建议按以下优先级排列：
1. 硬门槛不通过项（必须立即修复）
2. 0 分项（完全缺失，投入产出比最高）
3. 低于 60% 的项（有明显缺陷）
4. 低于 80% 的项（可快速提升）

每条建议包含：
- 当前问题
- 具体改进动作（可操作，不是空话）
- 预期提升分数

## 注意事项

- 评分必须以实际找到的证据为准，不能凭猜测给分
- 找不到证据的项给 0 分，但在评语中标注「未找到相关文件」
- 四、五两个维度涉及过程记录和团队协作，可能需要额外文件或现场演示，如果项目中没有相关记录，标注「需现场补充」
- 扫描时注意 docs/、tests/、skills 配置等多个目录
- 评分报告输出到对话，不写入文件（除非用户明确要求）
