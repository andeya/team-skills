# SDD 规格：Team Skills 自评分改进

> specAgent 产出 | 2026-06-23

## 一、背景与动机

Team Skills 项目定义了完整的评分体系（team-score），但项目自身尚未经过正式评分。通过自评分并迭代改进到满分，可以：
1. 验证评分体系本身的可执行性
2. 证明项目自身遵循了它所定义的规范
3. 为潜在用户展示"从评分到满分"的完整案例

## 二、业务规则

- **MUST** 基于实际证据评分，不凭猜测给分
- **MUST** 7 项硬门槛全部通过才能声明满分
- **MUST** 每个二级验收项有可检查的证据
- **SHOULD** 按优先级修复（硬门槛 > 0 分项 > 低于 60% > 低于 80%）

## 三、关键设计决策

| 决策 | 选择方案 | 拒绝方案 | 拒绝理由 |
|------|----------|----------|----------|
| 评分方式 | 5 子 Agent 并行扫描 | 单 Agent 顺序扫描 | 并行扫描效率更高，互不干扰 |
| 修复策略 | 先修复 D5（最快见效），再修复 D2 | 按维度顺序修复 | D5 修复只需补章节，D2 需要创建新文件 |
| 满分标准 | 全部维度满分 | 总分 ≥ 90 | 用户要求"真正获得满分" |

## 四、数据流

```
用户请求评分
  → Agent 1: 资产扫描 → 证据汇总
  → Agent 2: 任务规划扫描 → 证据汇总
  → Agent 3: 质量保障扫描 → 证据汇总
  → Agent 4: 使用过程扫描 → 证据汇总
  → Agent 5: 团队协作扫描 → 证据汇总
  → 硬门槛判定 → 逐项评分 → 总分 → 改进建议
  → 修复非满分项 → 重新评分 → 直到满分
```

## 五、输入规格

| 参数 | 类型 | 必填 | 约束 | 默认值 | 示例 |
|------|------|------|------|--------|------|
| 项目路径 | string | 是 | 必须是 team-skills 仓库根目录 | — | `/home/andeya/workpath/src/team-skills` |

## 六、输出规格

| 场景 | HTTP 状态 | 输出结构 | 示例 |
|------|-----------|----------|------|
| 评分完成 | — | 评分报告（对话中） | 见 team-score SKILL.md |
| 满分达成 | — | 评分报告 + 满分确认 | 全部维度满分 |

## 七、边界条件

| 条件 | 预期行为 | 测试优先级 |
|------|----------|-----------|
| 所有维度满分 | 声明满分达成 | P1 |
| 部分维度非满分 | 输出改进建议，继续迭代 | P1 |
| 硬门槛不通过 | 标记不通过，优先修复 | P0 |
