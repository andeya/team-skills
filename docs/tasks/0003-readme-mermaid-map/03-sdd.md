# SDD 规格：README Mermaid 使用地图

> specAgent 产出 | 2026-06-23

## 一、背景与动机

README 目前有文字描述的 Skill 清单和架构图，但缺少一张让用户**快速理解 Skill 关系和用法**的地图。用户需要知道：

- 什么场景该用哪个 Skill？
- Skill 之间谁依赖谁？
- 是自动跑还是手动调？

一张好的 Mermaid 流程图能替代数百字说明，让用户 10 秒掌握全貌。

## 二、业务规则

- **MUST** 覆盖全部 12 个 Skill（含 using-team-skills）
- **MUST** 区分"用户主动调用"和"编排器自动调度"两种路径
- **MUST** 标注 Skill 之间的前后依赖关系（谁产出给谁消费）
- **SHOULD** 从用户场景出发（需求模糊/明确/已有规格/已有实现等）
- **SHOULD** 与 README 现有内容风格协调

## 三、关键设计决策

| 决策 | 选择方案 | 拒绝方案 | 拒绝理由 |
|------|----------|----------|----------|
| 图的数量 | 1 张大图 | 多张小图 | 一张图能展示完整关系，用户不需要切换视角 |
| 布局方向 | 自上而下（Top-Down） | 从左到右 | 符合阅读习惯，与现有架构图风格一致 |
| 用户调用表示 | 虚线箭头 + 用户图标 | 颜色区分 | 虚线更直观，GitHub Markdown 支持有限 |
| 分组方式 | 按阶段分组（需求→规格→实现→测试→审查→交付） | 按角色分组 | 用户更关心"下一步做什么"而非"谁在做" |

## 四、数据流

```
用户（顶部入口）
  ├── 需求模糊 ──→ team-brainstorm ──→ design-brief
  ├── 需求明确 ──→ team-spec ──→ SDD 规格
  ├── 已有规格 ──→ team-impl ──→ 代码
  ├── 已有实现 ──→ team-test ──→ 测试报告
  ├── 已有代码+测试 ──→ team-review ──→ 审查报告
  ├── 收到审查反馈 ──→ team-feedback ──→ 修改
  ├── 遇到 bug ──→ team-debug ──→ 修复
  ├── 声明完成 ──→ team-verify ──→ 验证通过
  ├── 实现完成 ──→ team-finish ──→ 合并/PR
  ├── 需要完整流水线 ──→ team-orchestrator（自动调度上述流程）
  └── 评估成熟度 ──→ team-score ──→ 评分报告
```

## 五、输入规格

| 参数 | 类型 | 必填 | 约束 | 默认值 | 示例 |
|------|------|------|------|--------|------|
| README 路径 | string | 是 | 必须是项目根目录 | — | `README.md` |

## 六、输出规格

| 场景 | 输出结构 | 示例 |
|------|----------|------|
| 修改完成 | 更新后的 README.md | Mermaid 流程图插入在 Skill 清单之前 |

## 七、边界条件

| 条件 | 预期行为 | 测试优先级 |
|------|----------|-----------|
| Mermaid 语法错误 | GitHub 不渲染，需验证语法 | P0 |
| 图过于复杂 | 拆分或简化，确保可读性 | P1 |

## 八、异常场景

| 异常 | 处理方式 |
|------|----------|
| Mermaid 渲染不兼容 | 保留 ASCII 架构图作为 fallback |
