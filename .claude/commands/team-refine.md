# /team-refine — Skill 质量精炼（对抗审计 + 执行打磨）

## 功能

对 `skills/_team-rules/skill-spec.md`（语言规范）与 `skills/*/SKILL.md`（全部消费方）进行双向对抗审计，同时按编排流程顺序逐个打磨每个 Skill 的 LLM 执行质量。每轮发现全部问题 → 修复 → 验证，循环直到收敛或达到轮次上限。

## 红线原则

以下两条红线贯穿每轮每个维度，违反即为缺陷：

1. **零阅读门槛**：skill-spec 定义的语法必须让 LLM 即使从未读过 skill-spec.md 也能理解用该语法写出的 SKILL.md 的含义。需要查阅规范才能理解的构造 = 设计缺陷。
2. **关键词全大写粗体英文**：skill-spec 定义的所有关键词（参见 skill-spec.md §2 关键词表）在执行上下文中必须 `**全大写粗体**`。表达式运算符（EXISTS/NOT_EXISTS 等）在反引号内使用纯大写。

## 参数

- `$ARGUMENTS`：最大轮次数（正整数，默认 5）

## 审计范围

- **规范文件**：`skills/_team-rules/skill-spec.md`
- **消费方文件**：`skills/*/SKILL.md`（所有 Skill 目录）
- **共享规则文件**：`skills/_team-rules/*.md`

## 审计维度（15 维度）

每轮必须覆盖全部维度，不可选择性跳过。

### A 组：Spec ↔ Skills 一致性（D1-D10）

| # | 维度 | 检查要点 |
|---|------|---------|
| 1 | 关键词格式 | 执行上下文中关键词是否 `**全大写粗体**`；引用块/表格中是否 `` `反引号` `` |
| 2 | 控制流完备性 | IF 有 ELSE/fallback、MATCH/RESOLVE 有 `*default*`/`*none*` 兜底、REPEAT 有 exhausted 处理 |
| 3 | Spec ↔ 用法一致性 | spec 定义能否覆盖 skill 中实际出现的所有语法构造 |
| 4 | 跨文件一致性 | 约定变量名统一（`verify_cmd`/`slug`/`mode`）、RESOLVE 链顺序一致 |
| 5 | 结构合规性 | SKILL.md 是否包含 CLAUDE.md §二 定义的 12 个必要章节 |
| 6 | 表达式语言 | 反引号表达式使用英文运算符，不混用中文 |
| 7 | Spec 覆盖率 | skill-spec 定义的每个构造是否至少被 1 个 skill 使用 |
| 8 | 零阅读门槛 | 模拟未读过 spec 的 LLM，每个语法构造是否自解释 |
| 9 | 引用完整性 | GOTO 目标存在、ASSERT 有 fallback、跨文件引用路径正确 |
| 10 | 关键词语义唯一 | 同一关键词不在不同场景表达不同含义 |

### B 组：LLM 执行质量（D11-D15）

| # | 维度 | 检查要点 |
|---|------|---------|
| 11 | 语义准确性 | ASSERT 条件与意图一致、EXEC 命令无交互阻塞、完成状态使用四态协议 |
| 12 | Intent blocks | 每个 Step/Phase 标题后紧跟 `> intent`（无 bare `>` 空行）、内容与步骤行为一致 |
| 13 | 共享规则引用 | 不内联重复 `_team-rules/` 已定义内容、不引用 team-skills CLAUDE.md 概念定义 |
| 14 | 自检门禁完整性 | GATE 含产出文件检查 + 占位符检查 + Iron Law 检查 + 四态声明 + 至少 1 条对抗性自问 |
| 15 | 集成关系准确性 | 上下游 Skill 名称正确、REQUIRED/推荐标注与实际依赖一致 |

### 误报排除

以下不是缺陷：

- 概念名称（YAGNI 等）的粗体标注——不是关键词
- 反引号代码块中的原样大写——代码环境不需要粗体
- `**required**`/`**optional**` 小写粗体——输入标注约定
- 声明性章节（角色定位、推理检查点）中的非执行引用块——无需 intent 格式

## 单轮执行协议

### Step 0：共享规则同步检查

每轮首先扫描 `skills/_team-rules/*.md`：检查残留的 `CLAUDE.md §` 概念引用（用户项目引用除外）、跨文件交叉引用是否正确。发现问题先修复共享文件，再进入 Skill 审计。

### Step 1：读取全部文件

读取 skill-spec.md 和所有 SKILL.md（按打磨顺序）。每轮必须重新读取——上一轮修复可能引入新问题。

### Step 2：逐 Skill 全维度扫描 + 修复

按打磨顺序逐个处理 13 个 Skill，每个覆盖全部 15 维度。发现问题立即记录：

- 文件路径 + 行号
- 维度编号（D1-D15）
- 具体问题描述 + 修复方案

扫描完成后按清单逐一修复：

- 用法不合规 / 执行质量问题 → 修改 skill
- 规则缺失或不合理 → 修改 spec
- 红线违反 → 优先改 spec（让语法自解释）
- 0 个问题 → 标记"干净"

### Step 3：验证修复 + 输出报告

重新读取被修改的文件，验证修复正确且未引入新问题。输出本轮报告：

```
## Round {N}/{max}

| Skill | A组 | B组 | 合计 | 状态 |
|-------|-----|-----|------|------|

问题明细（含行号、维度、修复内容）

本轮：{total} 个问题 | 累计修复：{cumulative}
```

## 迭代控制

- **收敛退出**：连续 2 轮 0 问题 → 提前结束
- **轮次上限**：达到参数值（默认 5）→ 停止
- **死循环检测**：连续 3 轮问题集合完全相同 → 标记"无法自动修复"，停止

## 最终产出

```
# 精炼总结

- 轮次：{N}/{max}，收敛于 Round {N}
- 累计：{total} 个问题（A组 {a} / B组 {b}），修复 {fixed}
- 修改文件：{files} 个
- 维度分布：D1({n}) D2({n}) ... D15({n})

| Skill | 最终问题数 | 累计修复 | 状态 |
|-------|-----------|---------|------|
```
