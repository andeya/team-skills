# /team-refine — 改进飞轮（对抗审计 + 执行打磨）

## 功能

对 `CLAUDE.md` §2.7（Skill Spec）与 `skills/*/SKILL.md`（全部消费方）进行双向对抗审计，同时按编排流程顺序逐个打磨每个 Skill 的 LLM 执行质量。每轮发现全部问题 → 修复 → 验证，循环直到收敛或达到轮次上限。

## 红线原则

以下红线贯穿每轮每个维度，违反即为缺陷：

1. **零阅读门槛** — 语法构造必须让 LLM 即使从未读过规范文档也能自然理解。需要查阅规范才能理解的构造 = 设计缺陷。
2. **关键词全大写粗体英文** — 关键词在执行上下文中必须 `**全大写粗体**`。表达式运算符在反引号内使用纯大写。
3. **缩写即缺陷** — 所有约定的标识必须使用自解释的全大写英文。`**ASK_HUMAN**` 是标杆——LLM 看到就知道"问人类"。`**H3**` 是反面教材——LLM 猜不出含义。

## 参数

- `$ARGUMENTS`：最大轮次数（正整数，默认 5）

## 审计范围

- **规范文件**：`CLAUDE.md` §2.7（Skill Spec）
- **消费方文件**：`skills/*/SKILL.md`（所有 Skill 目录）
- **共享规则文件**：`skills/_team-rules/*.md`
- **模板文件**：`skills/*/references/*.md`

## 审计维度（15 维度）

每轮必须覆盖全部维度，不可选择性跳过。每个维度是一种**发现方法**，不是具体问题清单——方法不变，发现的问题随代码库状态变化。

### A 组：Spec ↔ Skills 一致性（D1-D10）

| # | 维度 | 发现方法 |
|---|------|---------|
| 1 | 关键词格式 | grep 所有 `**粗体大写**` 出现，对照关键词表：表中的必须加粗，表外的不能加粗。反引号内的关键词不加粗 |
| 2 | 控制流完备性 | 遍历每个 IF/MATCH/RESOLVE/REPEAT，检查是否有未覆盖的出口（无 ELSE/无 `*DEFAULT*`/无 `*REPEAT_EXHAUSTED*`） |
| 3 | Spec ↔ 用法一致性 | 列出 spec 定义的所有构造，对照 SKILL.md 中实际出现的所有构造。spec 有但 skill 没用到的 → 冗余；skill 用了但 spec 没定义的 → 缺口 |
| 4 | 跨文件一致性 | 对比所有 SKILL.md 中相同语义的变量名（verify_cmd/slug/mode 等），检查命名和 RESOLVE 链顺序是否一致 |
| 5 | 结构合规性 | 对照 CLAUDE.md §二 的 12 章节清单（ROLE/IRON_LAW/QUALITY/INPUT/STEPS/OUTPUT_TEMPLATE/SELF_CHECK/COMPLETION/STOP_SIGNALS/INTEGRATION/NEXT），检查每个 SKILL.md 是否包含。有文件产出的 Skill 应有 OUTPUT_TEMPLATE 章节或内联骨架；无文件产出或有内联骨架的可省略 |
| 6 | 表达式语言 | grep 反引号内的表达式，检查运算符是否使用英文（EXISTS/NOT_EMPTY/CONTAINS 等），是否有中文运算符混入 |
| 7 | Spec 覆盖率 | 关键词表中每个构造，至少被 1 个 SKILL.md 使用。未被使用的构造 → 冗余或示例不足 |
| 8 | 零阅读门槛 | 扫描所有非英文单词的标识（缩写、数字编号、中文标签），对每个标识问：LLM 第一次看到能猜出含义吗？猜不出的 → 设计缺陷。特别关注系统提示词和关键词表中的标识 |
| 9 | 引用完整性 | 追踪每个 GOTO/ROLLBACK/ROUTE 的目标是否存在。每个 ASSERT 是否有 fallback 分支。跨文件引用路径是否正确 |
| 10 | 关键词语义唯一 | 同一关键词在不同 SKILL.md 中是否表达相同含义。同一含义是否用了不同关键词 |

### B 组：LLM 执行质量（D11-D15）

| # | 维度 | 发现方法 |
|---|------|---------|
| 11 | 语义准确性 | 逐条执行每个 ASSERT/EXEC/IF，检查条件是否与意图一致、命令是否可执行、完成状态是否符合四态协议 |
| 12 | Intent blocks | 每个 `### Phase/Step` 标题后第一行是否为 `>` 意图行。检查是否有 bare `>` 空行。意图内容是否与步骤行为一致 |
| 13 | 共享规则引用 | grep 所有 `_team-rules/` 引用，检查是否有内联重复（已定义在共享文件中的内容被直接写在 SKILL.md 中） |
| 14 | SELF_CHECK 完整性 | 每个 GATE 检查：是否包含产出文件检查 + 占位符检查 + IRON_LAW 检查 + 四态声明 + 至少 1 条对抗性自问 |
| 15 | INTEGRATION 准确性 | 对照编排流程图，检查每个 Skill 的上下游声明是否与实际依赖一致。REQUIRED/推荐标注是否准确 |

### 误报排除

以下不是缺陷：

- 概念名称（YAGNI 等）的粗体标注——不是关键词
- 反引号代码块中的原样大写——代码环境不需要粗体
- `**required**`/`**optional**` 小写粗体——输入标注约定
- 声明性章节（ROLE、推理检查点）中的非执行引用块——无需 intent 格式
- 文件名/领域专有名词——在文件路径或文件内容引用中出现时，不是语法构造，不要求自解释

## 单轮执行协议

### Step 0：共享规则同步检查

扫描 `skills/_team-rules/*.md` 和 `skills/*/references/*.md`：检查跨文件交叉引用是否正确、共享规则文件之间是否有矛盾。发现问题先修复共享文件，再进入 Skill 审计。

### Step 1：批量扫描（全维度 × 全文件）

按维度批量扫描，而非按 Skill 逐个扫描。每个维度是一种发现方法，一次性应用于所有文件：

```
FOR dimension IN D1..D15:
  应用该维度的发现方法到 13 个 SKILL.md + _team-rules/*.md + references/*.md
  记录所有违规（文件路径 + 行号 + 维度 + 问题描述 + 修复方案）
```

同一维度的问题模式相似，批量发现比逐个 Skill 切换更快。

### Step 2：按清单修复

扫描完成后，按清单逐一修复：

- 用法不合规 / 执行质量问题 → 修改 skill
- 规则缺失或不合理 → 修改 spec
- 红线违反 → 优先改 spec（让语法自解释）
- 0 个问题 → 标记"干净"

### Step 3：验证修复 + 输出报告

重新读取被修改的文件，验证修复正确且未引入新问题。特别检查：

- 修复是否破坏了 Markdown 结构（mermaid 图、表格、代码块）
- 修复是否与跨文件引用一致
- 本轮修复的问题是否在另一维度视角下仍然有问题

输出本轮报告：

```
## Round {N}/{max}

| Skill | A组 | B组 | 合计 | 状态 |
|-------|-----|-----|------|------|

问题明细（含行号、维度、修复内容）

本轮：{total} 个问题（A组 {a} / B组 {b}）| 累计修复：{cumulative}
```

### Step 4：team-score 满分校验

基于 Step 1 已读取的 `skills/team-score/SKILL.md`，以 team-score 的 **5 维度 + 7 硬门槛** 为检查表，反向审计全部 Skill 流水线——验证一个项目在完整执行 team-skills 编排流程后，能否在每个评分项上拿到满分。

#### 4.1 硬门槛覆盖检查

逐条验证 7 个硬门槛对应的产出是否在某个 Skill 的产出文件模板中被**显式要求**：

| # | 硬门槛 | 应覆盖 Skill |
|---|--------|-------------|
| 1 | AI 任务规划文档 | team-spec |
| 2 | AI 修改边界 | team-spec / team-impl |
| 3 | 测试/验证手段 | team-test / team-verify |
| 4 | 预设测试通过 | team-impl / team-finish |
| 5 | 协作资产非模板化 | team-review / team-finish |
| 6 | 风险与未覆盖项说明 | team-spec / team-review |
| 7 | 关键决策说明 | team-brainstorm / team-spec |

若某硬门槛在流水线中无 Skill 产出对应证据 → 记录为**满分缺口**。

#### 4.2 评分维度覆盖检查

对 team-score 5 个维度的每个评分子项：

1. 定位该子项要求的**证据类型**（文件/代码/命令输出/截图）
2. 追溯到流水线中哪个 Skill 负责产出该证据
3. 检查该 Skill 是否在产出文件模板或执行步骤中**显式指导**生成此证据
4. 缺失 → 记录为满分缺口，附修复方案（补充产出模板 / 增加步骤 / 强化自检门禁）

**排除项**（流水线无法覆盖，不计为缺口）：

- D4.5 "评委自定"：依赖现场答辩，team-score 已规定"无答辩环节时此项不评分"
- D5 各项在单人项目模式下的适配解读：team-score 已内置适配规则

#### 4.3 缺口修复

按优先级修复发现的满分缺口：

- **P0**：硬门槛无覆盖 → 在对应 Skill 中增加产出要求或执行步骤
- **P1**：评分子项得分 < 满分 → 强化对应 Skill 的模板/门禁
- **P2**：证据格式不匹配 → 对齐 Skill 产出格式与 team-score 期望的证据格式

#### 4.4 输出校验报告

```
### team-score 满分校验

| 维度 | 子项数 | 满分覆盖 | 缺口数 | 状态 |
|------|-------|---------|-------|------|
| D1 AI协作资产 | {n} | {covered}/{n} | {gap} | ✅/❌ |
| D2 任务规划 | {n} | {covered}/{n} | {gap} | ✅/❌ |
| D3 交付质量 | {n} | {covered}/{n} | {gap} | ✅/❌ |
| D4 过程复盘 | {n} | {covered}/{n} | {gap} | ✅/❌ |
| D5 团队协作 | {n} | {covered}/{n} | {gap} | ✅/❌ |

硬门槛：{passed}/7 覆盖
满分缺口：{total_gaps} 个（P0: {p0} / P1: {p1} / P2: {p2}），已修复 {fixed}
```

满分校验未通过（存在未修复缺口）时，该轮不视为"0 问题"，不触发收敛退出。

### Step 5：普遍特征提取与规范迭代

> WHY：仅靠 Spec → Skills 的向下审计会遗漏 Skills 实践中涌现的隐性共识。Skills 的普遍使用模式是 Spec 迭代的最佳信号——LLM 天然倾向于使用自然、高效的表达方式，这些方式如果不在 Spec 中，说明 Spec 落后于实践。

扫描全部 SKILL.md + _team-rules/*.md + references/*.md，从 Skills 的实际使用模式中提取共性，驱动 Spec 自底向上迭代。

#### 5.1 隐性共识检测

对比 Spec 关键词表与所有 SKILL.md 的实际使用：

- 所有（或 >50%）Skill 都使用但 Spec 未定义的构造 → **必须**加入 Spec
- 所有（或 >50%）Skill 都遵循但 Spec 未规定的约定 → **必须**纳入 Spec
- 超过 3 个 Skill 使用相同模式但 Spec 未覆盖 → **应**评估是否加入 Spec

> TRAP：不要因为"这个模式不够优雅"而拒绝纳入 Spec。如果 LLM 天然倾向于使用它，说明它有合理性。先纳入，再优化。

#### 5.2 变体分歧检测

同一语义在不同 Skill 中的表达差异，是 Spec 规范力度不足的信号：

- **命名差异**：同一概念在不同 Skill 中使用了不同名称（如 `verify_cmd` vs `test_cmd`、`slug` vs `project_name`）
- **结构差异**：同一章节在不同 Skill 中结构不一致（如 SELF_CHECK 的条目顺序、COMPLETION 的格式）
- **格式差异**：同一构造在不同 Skill 中格式不同（如 `**RESOLVE**` 链的缩进风格、`**GATE**` 的条目排列）
- → 统一到 Spec，作为规范约束

> SIGNAL：变体分歧越多，说明 Spec 对该区域的约束越弱。优先处理出现频率最高的分歧。

#### 5.3 冗余模式检测

检查所有 Skill 中重复书写的内容，驱动共享化：

- 所有 Skill 都包含的相同段落 → 提取到 `_team-rules/` 共享文件
- 多个 Skill 使用相同模板结构 → 提取到 `references/` 模板文件
- 重复的 IRON_LAW 或 STOP_SIGNALS 模式 → 抽象为共享规则生成器
- 同一段说明文字在 N 个 Skill 中出现 → 每多一个副本就是多一个维护点

> GOOD：`verification-protocol.md` 被所有 Skill 引用而非内联 → 正确的共享化
> BAD：每个 Skill 各自写一段 "四态协议说明" → 应引用 `four-state-protocol.md`

#### 5.4 死码检测

Spec 定义了但所有 Skill 都未使用的构造：

- 评估该构造是否仍有价值 → 从 Spec 移除或增加示例用法
- 评估是否因文档不清晰导致无人使用 → 改进文档示例
- 评估是否因 LLM 能力已覆盖无需显式规定 → 从 Spec 移除

> SIGNAL：连续 3 轮 refine 都无人使用的构造 → 优先移除候选。

#### 5.5 规范迭代

基于 5.1-5.4 的发现，更新 `CLAUDE.md` §2.7（Skill Spec）：

- 新增隐性共识到关键词表或格式规范
- 统一变体分歧，更新规范约束
- 提取冗余模式到共享文件
- 移除或补充死码

### Step 6：方法论总结与自我改进

> WHY：没有方法论总结的 refine 是低水平重复。每轮积累的"什么方法有效、什么方法无效"是改进 refine 流程本身的关键输入。

总结本轮 refine 过程的有效方法，驱动 refine 流程本身的迭代。这是元认知层——不仅改进 Skills，还改进"改进 Skills 的方法"。

#### 6.1 有效方法评估

回顾本轮各维度的产出效率：

- **数量**：每个维度发现了多少问题
- **质量**：哪些维度发现了最关键的问题（红线违反、结构性缺陷）
- **空白**：哪些维度未发现问题 → 判断是方法本身无效还是确实无问题
- **效率排名**：按"问题数/耗时"排序，标记最高效和最低效的维度

> TRAP：某个维度发现 0 问题不意味着该维度无用——可能说明该区域已经成熟。连续 3 轮 0 问题才是移除候选。

#### 6.2 模式库持久化

将本轮提取的跨文件模式写入持久化存储：

- 新发现的共识模式 → 更新 `CLAUDE.md` 或 `_team-rules/`
- 统一后的命名规范 → 更新 `CLAUDE.md` §2.7
- 抽象出的共享规则 → 创建或更新 `_team-rules/` 文件
- 本轮最佳实践 → 写入 `team-refine.md` 自身（自我改进）

#### 6.3 流程改进建议

基于本轮经验，提出对 refine 流程本身的改进：

- 审计维度是否需要增删改
- 扫描方法是否需要优化（如新增 grep 模式、自动化脚本）
- 执行协议是否需要调整（如步骤顺序、检查深度、轮次控制）
- 本轮遇到的阻碍或效率瓶颈（如文件过大、重复劳动）
- 是否可以将部分审计维度自动化

#### 6.4 输出方法论报告

```
### 模式提取与自我改进

| 类别 | 发现数 | 已处理 | 未处理 |
|------|-------|-------|-------|
| 隐性共识 | {n} | {y} | {n-y} |
| 变体分歧 | {n} | {y} | {n-y} |
| 冗余模式 | {n} | {y} | {n-y} |
| 死码 | {n} | {y} | {n-y} |

有效方法排名：1. {维度} ({n} issues) 2. {维度} ({n} issues) ...
流程改进建议：{n} 条
```

## 迭代控制

- **最小轮次**：至少执行 3 轮（即使前 2 轮已收敛）
- **收敛退出**：连续 2 轮 D1-D15 审计 0 问题 **且** team-score 满分校验 0 缺口 **且** 模式提取无未处理项 → 提前结束（但不少于 3 轮）
- **轮次上限**：达到参数值（默认 5）→ 停止
- **死循环检测**：连续 3 轮问题集合（含满分缺口和模式提取结果）完全相同 → 标记"无法自动修复"，停止
- **回归检测**：上一轮已修复的问题在本轮重新出现 → 标记"回归"，修复方案需加强（如加自检门禁而非仅改文件）
- **自我改进**：每轮 Step 6 的方法论总结结果**必须**应用于下一轮 refine 流程。如果连续 2 轮流程改进建议完全相同且未执行 → 标记"自我改进停滞"，触发 ASK_HUMAN
