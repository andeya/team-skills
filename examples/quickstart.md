# 快速开始：你的第一个 Team Skills 流水线

本指南带你从零开始走完一个完整的 Team Skills 流水线。你会看到一条模糊的需求如何变成经过验证、审查和评分的交付物。

## 前置条件

- 已安装 Claude Code 或 Cursor
- 已安装 Team Skills（`/team-setup`）

## 第一步：从模糊需求开始

```bash
/team-orchestrator 实现一个用户注册功能
```

编排器会依次执行：

1. **H1**：向你确认是否理解目标
2. 调度 `team-spec` 创建完整的 SDD 规格
3. **H2**：请你审查并确认规格方案
4. 调度 `team-impl` 进行 TDD 实现
5. 调度 `team-test` 进行测试覆盖审计
6. 调度 `team-review` 进行代码审查
7. **H4**：向你交付最终成果等待验收

## 第二步：审查规格（H2）

规格 Agent 会产出类似如下的目录结构：

```
docs/tasks/0001-user-registration/
├── 01-plan.md          # 任务规划（目标、分期、预算）
├── 02-context.md       # 术语表、引用、排除项
├── 03-sdd.md           # 完整的七部分 SDD
├── 04-boundary.md      # 允许和禁止的修改范围
├── 05-risk.md          # 风险评估和验证计划
└── prompt-template.md  # 可复用的提示词模板
```

在 H2 阶段，重点审查 SDD 的以下内容：

- **业务规则**：MUST/SHOULD/MAY 的优先级是否正确？
- **边界条件**：边缘场景是否覆盖？
- **设计决策**：是否同意选型方案？

## 第三步：观察 TDD 执行

实现 Agent 运行红-绿-重构循环：

```
RED:     写一个会失败的测试
GREEN:   写最简代码让测试通过
REFACTOR: 优化代码质量
COMMIT:   git commit 提交
```

每个循环都记录在 `06-tdd-log.md` 中。你可以看到什么被测试、什么失败了、如何修复的。

## 第四步：测试审计

测试 Agent 运行四维测试矩阵：

| 维度 | 检查内容 |
|------|----------|
| 功能 | SDD 中的正常路径场景 |
| 边界 | 边缘情况、空值、极值 |
| 异常 | 错误处理、非法输入 |
| 代码分支 | 分支覆盖、条件覆盖 |

如果测试失败，Agent 会自动回退到实现 Agent，并附带具体的失败上下文。

## 第五步：审查报告

审查 Agent 产出五维审查报告：

1. **正确性**：是否匹配 SDD？
2. **完整性**：是否覆盖所有场景？
3. **一致性**：是否遵循项目规范？
4. **清晰性**：代码是否可读？
5. **合规性**：是否遵守 Constitutional Rules？

## 第六步：评分

```bash
/team-score
```

输出示例：

✅ 验证协议已执行
✅ 有向图回退已配置
✅ 资产体系已建立
✅ 完成状态已定义

维度：
📁 资产沉淀：    21/25
